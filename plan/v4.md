## 1. v3 的真正亮点（值得保留的设计决策）

把 v3 的优点收敛成 4 条核心设计决策，后续不再讨论：

1. **角色分离原则**：normalize 只做无损表面标准化（L1），keyword 做语义抽象（L2），ctx_fingerprint 做结构约束（L3）。三层职责互不重叠。
2. **LLM 角色降级**：从"生成开放式 keyword"降级为"在向量召回的候选中做二选一/新建决策"，输出空间从无限收敛到有限。
3. **向量索引粒度选择**：对 keyword 而非 query 建立向量索引。规模差几个数量级，规避大规模 ANN 延迟。
4. **结构化 ctx_fingerprint**：拒绝内容哈希，使用 `(类型, 长度桶, 工具集, schema)` 四元组，且 schema 用"包含关系"匹配。

这四条是 v3 相对 v1/v2 的真正价值，v4 全部继承。

## 2. v3 的缺陷分析（按影响等级）

### 2.1 P0 缺陷（会在生产环境直接触发故障）

#### a) L1 缓存键缺少 ctx 维度，存在串数据风险

v3 §3 中：

```
[L2/L3 匹配成功] → 提升到 L1: SET apc:exact:<agent_id>:<task_sig> <tpl_id>
```

**问题**：`task_sig = sha256(normalize(query) + agent_id + tool_hash)` 不包含 ctx 信息，但 L3 命中本身依赖 ctx_fingerprint 筛选。这意味着：

```
Request A: query="计算流动比率", ctx={财务报告 PDF, 长文档}
  → L2/L3 匹配到 tpl_finance_long
  → 提升到 L1: apc:exact:agent:Q  →  tpl_finance_long

Request B: query="计算流动比率", ctx={表格数据, 短文档}  ← 不同 ctx
  → L1 命中（task_sig 相同！）→ 直接返回 tpl_finance_long
  → 但 tpl_finance_long 假设输入是长文档，模板适配失败或产生错误结果
```

**根因**：L1 设计假设"完全相同的请求"，但只看 query 字面，没看 ctx。

**修复**：L1 改为双键结构：

```python
# 当 L1 写入是 L2/L3 命中提升时，使用复合键
exact_key = f"apc:exact:{agent_id}:{task_sig}:{ctx_fp_hash}"

# 或更激进：L1 完全只服务于"重复请求"，不做提升
# 提升到 L1 的逻辑改为提升到一个独立的 hot_cache 层
```

推荐后者：L1 只缓存"字面+ctx 都完全相同"的请求，提升机制单独走 `apc:hot` 层，避免双键复杂度污染 L1 语义。

#### b) LLM 归一化的失败回退路径完全缺失

v3 §2.3 给出了归一化 prompt，但没有处理：

| 失败场景 | v3 行为 | 后果 |
|---------|--------|------|
| LLM API 超时/限流 | 抛异常 | 整个请求失败 |
| LLM 返回格式不符（带解释、含 emoji、超长） | 直接 setex 入缓存 | kw_cache 被污染，后续命中也是脏数据 |
| LLM 返回的 keyword 包含 query 中的实体名（如 "costco_revenue_2019"） | 直接入缓存 | keyword 碎片化，违背归一化初衷 |
| LLM 返回完全空字符串 | setex 空字符串 | L2 查询永远 miss |

**修复**：归一化必须有完整的失败回退链：

```python
async def normalize_keyword(query: str, candidates: list[str]) -> str:
    try:
        raw = await llm.invoke(NORMALIZE_PROMPT, timeout=2.0)
    except (TimeoutError, RateLimitError):
        # 回退 1：用最高相似度候选（如果存在且超过阈值）
        if candidates and top_score(candidates[0]) > 0.7:
            return candidates[0]
        # 回退 2：用 v2 的纯生成路径
        return await fallback_extract_keyword(query)

    # 输出格式校验
    keyword = sanitize_keyword(raw)  # 长度、字符白名单、词数
    if not keyword:
        return await fallback_extract_keyword(query)

    # 反幻觉校验：如果 LLM 声称"复用候选"但返回的字符串不在候选中，
    # 视为新建（降级为生成路径）
    if keyword not in candidates and not is_well_formed_new_keyword(keyword):
        return await fallback_extract_keyword(query)

    # 反实体注入：keyword 不应包含 query 中的具体实体
    if contains_query_entities(keyword, query):
        # 引导 LLM 重试一次，强调"不要包含具体实体"
        keyword = await retry_with_stricter_prompt(query, candidates)

    return keyword
```

#### c) Embedding 模型版本未管理

v3 §5.1 提到了模型预加载，但没说：

- 如果未来从 `all-MiniLM-L6-v2` 升级到 `all-MiniLM-L12-v2`（或本地训练版本），向量空间不兼容
- 已经写入 `apc:kw_meta` 的 embedding 是旧模型产物，新模型 query embedding 与之做余弦相似度无意义
- 升级后没有迁移策略，要么向量搜索全错，要么必须 flush 全部 kw_meta

**修复**：embedding 入库时记录模型指纹，启动时校验：

```python
EMBED_MODEL_NAME = "all-MiniLM-L6-v2"
EMBED_MODEL_VER  = "v1"  # 自定义版本号，模型/微调权重变化时手动 bump
EMBED_DIM        = 384

async def write_kw_meta(keyword: str, vec: np.ndarray):
    await redis.hset(f"apc:kw_meta:{keyword}", mapping={
        "embedding": vec.tobytes(),
        "model": EMBED_MODEL_NAME,
        "model_ver": EMBED_MODEL_VER,
        "dim": EMBED_DIM,
        "created_at": time.time(),
    })

async def search_keyword_index(query_vec, top_k=3):
    # 跳过版本不匹配的 keyword（视为索引外）
    valid = []
    for keyword in await all_keywords():
        meta = await redis.hgetall(f"apc:kw_meta:{keyword}")
        if meta[b"model_ver"].decode() != EMBED_MODEL_VER:
            continue
        valid.append((keyword, np.frombuffer(meta[b"embedding"], dtype=np.float32)))
    # ... KNN
```

并提供 `reindex_keywords` 后台任务：在版本不一致时异步重算所有 keyword 的 embedding。

### 2.2 P1 缺陷（不会立即崩溃但会逐步劣化命中率）

#### d) 多副本部署下的内存 KNN 一致性问题

v3 §6.2 推荐"内存 KNN（keyword < 5000 时），不引入 Redis Stack"。但没考虑多副本：

- 每个 API 进程在内存中维护一份 keyword embedding 副本
- 副本 A 收到 cache miss 写入新 keyword X → 写到 Redis
- 副本 B 不知道 X 已存在 → 内存里没有 X 的 embedding → 下次相似 query 来时召回不到 X → 又生成一个 X' → 碎片化

**v3 隐含假设**：每次 KNN 都从 Redis 拉所有 keyword embedding。但 5000 个 384 维 float32 = 7.5MB，每请求拉一次会成为瓶颈（带宽 + 反序列化）。

**修复方案（按代价递增）**：

| 方案 | 代价 | 一致性 |
|-----|-----|--------|
| 每次从 Redis 拉全量 | 高（每请求 ~7.5MB 传输） | 强一致 |
| 本地 LRU 缓存 + Redis pub/sub 失效推送 | 中（首次请求热加载，后续从内存） | 最终一致（窗口 < 1s） |
| 直接用 Redis Stack VECTOR | 低（Redis 端 KNN，结果传输小） | 强一致 |

**推荐**：默认采用 pub/sub 失效推送方案。当 keyword > 5000 或多副本数 > 4 时，切换到 Redis Stack。

```python
# 写入新 keyword 时广播
async def publish_keyword_added(keyword: str):
    await redis.publish("apc:kw_changes", json.dumps({
        "op": "add", "keyword": keyword, "ts": time.time()
    }))

# 各副本订阅
async def subscribe_kw_changes(local_index: dict):
    pubsub = redis.pubsub()
    await pubsub.subscribe("apc:kw_changes")
    async for msg in pubsub.listen():
        evt = json.loads(msg["data"])
        if evt["op"] == "add":
            vec = await fetch_kw_embedding(evt["keyword"])
            local_index[evt["keyword"]] = vec
        elif evt["op"] == "remove":
            local_index.pop(evt["keyword"], None)
```

#### e) sentence-transformer 同步 encode 在高并发下的 OOM 风险

v3 §2.3.b：

```python
loop = asyncio.get_event_loop()
return await loop.run_in_executor(None, model.encode, text)
```

**问题**：

- `None` 使用默认 ThreadPoolExecutor，默认 `min(32, cpu_count + 4)` 个线程
- 高并发下每个线程都在跑 PyTorch 推理，每次 encode 创建多个 ndarray
- 没有信号量限制，QPS 突增时会同时跑几十个 encode → CPU 100% + 内存涨
- sentence-transformer 不是 thread-safe（取决于 backend），并发 encode 在某些版本下会数据错乱

**修复**：用专用执行器 + 信号量限流 + 批处理：

```python
EMBED_EXECUTOR = ThreadPoolExecutor(max_workers=4, thread_name_prefix="embed")
EMBED_SEMAPHORE = asyncio.Semaphore(8)  # 同时最多 8 个 encode in-flight

# 微批处理：累积 5ms 内的 encode 请求一起跑
class EmbedBatcher:
    def __init__(self, batch_window_ms=5, max_batch=32):
        self._queue: list[tuple[str, asyncio.Future]] = []
        self._lock = asyncio.Lock()
        # ...
    async def encode(self, text: str) -> np.ndarray:
        fut = asyncio.get_event_loop().create_future()
        async with self._lock:
            self._queue.append((text, fut))
            if len(self._queue) >= self.max_batch:
                await self._flush()
        return await fut

    async def _flush(self):
        items = self._queue[:]
        self._queue.clear()
        texts = [t for t, _ in items]
        loop = asyncio.get_event_loop()
        vecs = await loop.run_in_executor(EMBED_EXECUTOR, model.encode, texts)
        for (_, fut), vec in zip(items, vecs):
            fut.set_result(vec)
```

批处理收益：sentence-transformer 在 batch=16/32 时单 token 吞吐比 batch=1 高 5-10 倍。

#### f) Top-K 候选数量与距离阈值未优化

v3 默认 `top_k=3`，没有阈值过滤。问题：

- keyword_index 只有 5 个 keyword 时，top_k=3 会把不相关的也塞给 LLM，prompt 噪声大
- 候选距离都很远（如 cosine_sim < 0.3）时，LLM 仍被诱导"复用"，可能错误归并
- 候选距离都很近（如所有候选 > 0.85）时，可能多个候选都该复用，但 LLM 只能选一个

**修复**：自适应候选构造：

```python
DIST_HIGH = 0.85   # 高度相似，强复用倾向
DIST_LOW  = 0.50   # 低于此阈值视为不相关，不进 prompt

async def build_candidates(query_vec, top_k=5):
    raw = await knn_search(query_vec, top_k=top_k)
    filtered = [(kw, score) for kw, score in raw if score >= DIST_LOW]

    # 边界情况
    if not filtered:
        return []  # 直接走"新建" 路径，跳过归一化 LLM 调用
    if filtered[0][1] >= DIST_HIGH and len(filtered) == 1:
        return None  # 仅一个高度匹配，直接复用，跳过 LLM
    if filtered[0][1] >= DIST_HIGH and filtered[0][1] - filtered[1][1] > 0.15:
        return None  # 第一名远超第二名，直接复用，跳过 LLM
    return filtered[:3]
```

注意：当 `build_candidates` 返回 `None`（直接复用）或 `[]`（直接新建）时，**完全省去 LLM 调用**，是 v3 没充分利用的优化点。

#### g) kw_cache 漂移的可观测性缺失

v3 §2.2 承认 kw_cache 在新 keyword 写入后会漂移，但只给了"缩短 TTL"的钝刀。没有度量这个漂移有多严重的指标。

**修复**：旁路打点 + 自适应 TTL：

```python
DRIFT_SAMPLE_RATE = 0.05  # 5% 的 kw_cache 命中走旁路验证

async def keyword_with_drift_check(query: str):
    cached = await redis.get(f"apc:kw_cache:{qhash(query)}")
    if cached:
        if random.random() < DRIFT_SAMPLE_RATE:
            # 旁路：跑一遍完整流程对比
            asyncio.create_task(_compare_drift(query, cached))
        return cached
    # ...

async def _compare_drift(query: str, cached_kw: str):
    fresh = await full_normalize_pipeline(query)
    if fresh != cached_kw:
        kw_cache_drift_total.labels(direction="changed").inc()
        # 可选：当前 query 的高频副本主动失效
    else:
        kw_cache_drift_total.labels(direction="stable").inc()
```

观察 `drift_changed / (drift_changed + drift_stable)` 比例，<5% 时 TTL 维持 3600s，>15% 时降到 300s。

### 2.3 P2 缺陷（设计完整性问题，不影响主路径）

#### h) keyword 治理（容量、别名、回收）缺失

`apc:tpl_idx:<keyword>` 是 set，v3 没说：

- 单个 keyword 下最多挂多少个 template_id？高频 keyword 下可能积累上百模板
- template 被全局 LRU 驱逐时，对应的 tpl_idx 是否清理？（v3 完全没提及）
- 两个 keyword 实际上是同义词怎么办？没有合并机制

**修复**：

```python
# 1. 单 keyword 容量上限（FIFO 或 LRU）
MAX_TPL_PER_KW = 32

# 2. template 删除时联动
async def evict_template(tpl_id: str, keyword: str):
    await redis.hdel("apc:tpl", tpl_id)
    await redis.srem(f"apc:tpl_idx:{keyword}", tpl_id)
    if not await redis.exists(f"apc:tpl_idx:{keyword}"):
        await redis.hdel(f"apc:kw_meta", keyword)
        await publish_keyword_removed(keyword)

# 3. keyword 别名（管理工具，不放主路径）
# apc:kw_alias  <old_keyword>  <canonical_keyword>
# 写入新 template 时检查别名表
async def resolve_alias(keyword: str) -> str:
    alias = await redis.hget("apc:kw_alias", keyword)
    return alias.decode() if alias else keyword
```

别名机制让运维有手段在线合并冗余 keyword（运维发现 "wc_ratio" 和 "working_capital_ratio" 应合并 → 设别名），不需要重启服务或清缓存。

#### i) 没有"从 LLM 决策中持续学习"的演进路径

v3 把 LLM 归一化作为永久组件。但每次决策都是一次标注：query Q → keyword K（复用或新建）。这些数据可以训练一个轻量分类器/检索器，长期降低 LLM 调用频率。

**演进路径建议**（v4 之后的方向）：

```
Phase 1（当前）: 100% LLM 决策
Phase 2: 收集 (query_embed, candidates_embed, decision) 三元组到 PG
Phase 3: 训练一个二分类器 / 排序模型，在置信度高时绕过 LLM
Phase 4: LLM 仅在分类器低置信度时被调用（Active Learning）
```

这不是 v4 必须实现的，但应该写进设计文档作为长期方向，且数据收集（Phase 2）应该现在就开始：

```python
async def log_normalization_decision(query, query_vec, candidates, decision):
    # 异步写入，不阻塞主路径
    asyncio.create_task(_write_pg(...))
```

#### j) 跨语言/混合语言场景未讨论

v3 全程默认英文。但生产环境很可能遇到：

- 纯中文 query：`"计算 Costco FY2019 的流动比率"`
- 中英混合：`"Calculate Costco 在 FY2019 的 working capital"`

**问题**：

- normalize 中的 NFKC 处理不了中英混写的语义
- all-MiniLM-L6-v2 主要在英文语料训练，对中文 keyword 召回质量较差
- LLM 归一化时，候选可能是英文 keyword，query 是中文，混淆判断

**修复方向**：

- 用多语言 embedding 模型（`paraphrase-multilingual-MiniLM-L12-v2`）
- keyword 标准化为英文（即使 query 是中文，LLM 归一化时引导输出英文 keyword）
- ctx_fingerprint 中加 `query_lang` 字段，跨语言 query 不匹配相同模板

### 2.4 P3 缺陷（小瑕疵）

#### k) 性能对比表的数字未注明测试条件
v3 §4.1 给出了 `<1ms / 5ms / 200ms` 等数字，没说硬件、模型、batch size。落地时需要重新基准测试，写进文档时应注明"依据：[基准测试报告链接]"或"经验估算"。

#### l) `adjacent_buckets` 的传递性未定义
v3 §2.5 中 `adjacent_buckets("short", "medium")` 可，"short" 与 "long" 不可。但 `adjacent_buckets("medium", "long")` 是否可？文档没明说，会导致实现者各做各的。建议明确：

```python
BUCKET_ORDER = ["short", "medium", "long"]
def adjacent_buckets(a: str, b: str) -> bool:
    ia, ib = BUCKET_ORDER.index(a), BUCKET_ORDER.index(b)
    return abs(ia - ib) <= 1
```

#### m) `extract_data_schema` 实现未给出
v3 §2.5 提到 schema 是"列名集合 / 顶层 key 集合"，但嵌套结构如何处理（深度多少？）？给出参考实现：

```python
def extract_data_schema(context, max_depth=2) -> frozenset[str]:
    """提取 context 的结构特征，忽略具体值。"""
    keys = set()
    def walk(obj, depth=0, prefix=""):
        if depth >= max_depth or not isinstance(obj, dict):
            return
        for k, v in obj.items():
            full = f"{prefix}.{k}" if prefix else k
            keys.add(full)
            if isinstance(v, dict):
                walk(v, depth + 1, full)
    walk(context)
    return frozenset(keys)
```

## 4. v4 关键代码示例（汇总修改后的核心节点）

```python
async def keyword_node_v4(state: AgentState) -> AgentState:
    query = state["query"]
    qhash = sha256(normalize(query).encode()).hexdigest()[:16]

    # 1. kw_cache 命中（带漂移采样）
    cached = await redis.get(f"apc:kw_cache:{qhash}")
    if cached:
        kw = cached.decode()
        if random.random() < DRIFT_SAMPLE_RATE:
            asyncio.create_task(check_drift(query, kw))
        return {**state, "keyword": kw}

    # 2. 向量召回 + 自适应短路
    query_vec = await embed_batcher.encode(query)
    candidates = await build_candidates(query_vec, top_k=5)

    if candidates is None:
        # 强复用短路
        kw = candidates_top.keyword
    elif candidates == []:
        # 强新建短路（候选都太远，直接生成）
        kw = await fallback_extract_keyword(query)
    else:
        # 正常归一化路径
        kw = await normalize_keyword(query, candidates)

    kw = sanitize_keyword(kw)

    # 3. 写入 kw_cache（自适应 TTL）
    ttl = current_kw_cache_ttl()  # 由漂移率动态调整
    await redis.setex(f"apc:kw_cache:{qhash}", ttl, kw)

    # 4. 记录决策（异步，不阻塞）
    asyncio.create_task(log_normalization_decision(
        query, query_vec, candidates, kw
    ))

    return {**state, "keyword": kw}
```

```python
async def cache_lookup_v4(state: AgentState) -> AgentState:
    agent_id   = state.get("agent_id", "default")
    tools_hash = state.get("tools_hash", "default")
    query      = state["query"]
    ctx        = state["context"]

    # L1: 精确匹配（task_sig + ctx_fp）
    task_sig = sha256(
        f"{normalize(query)}|{agent_id}|{tools_hash}".encode()
    ).hexdigest()[:16]
    ctx_fp = compute_fingerprint(ctx, state.get("tools", []), agent_id)
    ctx_fp_hash = sha256(json.dumps(ctx_fp, sort_keys=True).encode()).hexdigest()[:8]

    exact_key = f"apc:exact:{agent_id}:{task_sig}:{ctx_fp_hash}"
    if tpl_id := await redis.get(exact_key):
        if tpl := await load_and_validate_template(tpl_id):
            cache_hits_total.labels(layer="L1").inc()
            return _hit(state, tpl, layer="L1")

    # L2/L3: 走 keyword 路径
    state = await keyword_node_v4(state)
    candidates = await redis.smembers(
        f"apc:tpl_idx:{agent_id}:{state['keyword']}"
    )
    if not candidates:
        return {**state, "cache_hit": False}

    for tpl_id in list(candidates)[:MAX_CANDIDATES_TO_CHECK]:
        tpl = await load_and_validate_template(tpl_id)
        if tpl and ctx_compatible(tpl["ctx_fingerprint"], ctx_fp):
            # 提升到 hot 层（不污染 L1 语义）
            await redis.set(f"apc:hot:{agent_id}:{task_sig}:{ctx_fp_hash}",
                            tpl_id, ex=86400)
            cache_hits_total.labels(layer="L2_L3").inc()
            return _hit(state, tpl, layer="L2_L3")

    return {**state, "cache_hit": False}
```

## 5. v3 与 v4 差异对照表

| 维度 | v3 | v4 |
|------|----|----|
| L1 缓存键 | `task_sig` 单维 | `task_sig + ctx_fp_hash` 双维 |
| L1 提升机制 | L2/L3 命中 → 写 L1 | L2/L3 命中 → 写独立 hot 层，L1 仅服务真正重复请求 |
| LLM 归一化失败 | 抛异常或污染缓存 | 多级回退（候选短路 → 纯生成路径 → 重试） |
| Embedding 模型版本 | 不管理 | model_ver 入库 + 启动校验 + reindex 工具 |
| 多副本一致性 | 隐含假设单副本 | 本地缓存 + pub/sub 失效推送 |
| encode 并发 | 默认线程池 | 专用 executor + 信号量 + 微批处理 |
| 候选构造 | 固定 top_k=3 | 阈值过滤 + 强复用/强新建短路 |
| kw_cache TTL | 固定 600s（建议值） | 由漂移率动态调整（300s ~ 3600s） |
| keyword 治理 | 未涉及 | 容量上限 + 联动驱逐 + 别名机制 |
| 学习闭环 | 无 | 决策日志收集，为 Phase 2 模型化铺路 |
| 多语言 | 默认英文 | 多语言 embedding + ctx 加 query_lang |
| 失败可观测 | 一般指标 | 漂移率、归一化失败率、回退分布、版本不一致计数 |

## 6. 不需要改的地方（避免范围蔓延）

下面这些是工程师容易"顺手优化"但 v4 阶段**不应该**动的：

1. **不引入 Redis Stack VECTOR**：keyword < 5000 场景下，本地 KNN + pub/sub 已经够用，引入 Redis Stack 会增加部署复杂度（不同 Redis 镜像、模块加载）。等到规模真到了再切。
2. **不替换 sentence-transformer 为 OpenAI Embedding**：本地模型零 API 成本、零网络延迟、隐私可控。除非 keyword 数 > 50000 且本地推理成为瓶颈，否则不切。
3. **不上模糊 LLM 模板适配**：v3 已经在 L3 用结构化 ctx_fingerprint 兜住模板复用约束。再叠一层"用 LLM 判断模板是否适用"会破坏论文方法的成本优势。
4. **不在 v4 引入 query 级向量索引**：v3 明确拒绝了对 query 做向量匹配（规模过大、ANN 延迟）。v4 不要回头走老路。
5. **不重新设计 LangGraph 拓扑**：v2 的图修复（template_gen 路由 bug）已经覆盖了核心问题。v3/v4 的改动都在节点内部，拓扑不变。

## 附录：v1 → v4 演进总览

```
v1: 单层 LLM keyword 抽取
    问题: 漂移、每请求 LLM、无 ctx 约束、template_gen bug
            ↓
v2: 三层结构（exact / keyword+kw_cache / ctx_筛选）
    + 修 template_gen bug + 失败处理 + 多租户
    遗留: ctx_fingerprint 是内容哈希（无效）、keyword 仍 LLM 直接生成
            ↓
v3: 向量召回 + LLM 归一化决策
    + ctx_fingerprint 改结构化 + 命名规范化
    遗留: L1 缺 ctx 维、归一化失败回退缺、embedding 版本不管、多副本一致性、encode 并发未治理
            ↓
v4: P0 修补 + P1 健壮 + P2 演进路径
    新增: 决策日志收集（为长期降本铺路）+ 多语言支持
    边界: 不引入 Redis Stack / OpenAI Embedding / 模糊 LLM 适配（避免范围蔓延）
```

到 v4 为止，主路径设计可以认为收敛。后续演进的重点应转移到：
- 用收集到的归一化决策数据训练替代模型，逐步把 LLM 调用频率压到 0
- 推测性预生成（speculative pre-warming）：根据当前请求预测下一个可能的请求，后台预跑
- 跨 Agent 缓存共享：同一 keyword 在不同 Agent 间的模板互通条件研究
