# v4.md 架构 Review 与 v5 优化方案

> Review 日期：2026-05-09
> 范围：v4.md 全部内容（v3 补丁方案、P0-P3 缺陷分析、代码示例、落地节奏）
> 立场：v4 是一份质量较高的工程化补丁，但部分修正引入了比原问题更复杂的机制。v5 做减法。

---

## 0. 总体结论

| 维度 | v4 评分 | 说明 |
|------|---------|------|
| 缺陷识别准确度 | ★★★★☆ | P0 三项都打对了点；P1 多数合理，但 encode 并发问题有点"过度防御" |
| 修复方案质量 | ★★★☆☆ | 归一化回退链、自适应候选构造很好；**L1/hot 双键拆分是过度设计** |

v4 的问题不是方向错误，而是**修正热情过高**——部分 P1 问题用简单的方案就能解决，v4 选了更复杂的那个。

## 1. v4 的真实优点（值得保留）

### 1.1 范围控制意识

v4 §6 明确列出 5 个"不应该改"的点：

- 不引入 Redis Stack VECTOR
- 不替换本地 embedding 为 OpenAI Embedding
- 不上模糊 LLM 模板适配
- 不引入 query 级向量索引
- 不重新设计 LangGraph 拓扑

这种"工程师容易顺手优化但实际不该动"的清单，在实际项目里比新功能设计更有价值。v5 全部继承，再加一条：**不对 keyword 做 LLM 驱动的语义聚类/合并（走别名机制即可）。**

### 1.2 LLM 归一化的失败回退链

v4 §2.1.b 的多级回退设计扎实：

```
LLM 超时/限流 → 候选短路（top_score > 0.7）→ fallback_extract_keyword
输出格式异常 → sanitize_keyword → fallback_extract_keyword
反幻觉校验   → 声称复用但不在候选列表 → fallback
反实体注入   → retry_with_stricter_prompt
```

这个回退链覆盖了 LLM 在生产中最常见的 4 种失败模式，且每一级都有明确的触发条件和降级目标。

### 1.3 自适应候选构造的短路优化

v4 §2.2.f 引入两个短路：

- **强复用短路**：只有一个候选且 cosine_sim ≥ 0.85，或第一名远超第二名（差距 > 0.15）→ 直接复用，跳过 LLM
- **强新建短路**：所有候选 cosine_sim < 0.50 → 直接走 fallback_extract_keyword，跳过归一化 LLM

这利用了 v3 没利用的信息：向量距离本身就是置信度信号。在高置信场景下省 LLM 调用，低置信场景才引入 LLM 判断。这是正确的分工。

### 1.4 漂移可观测性

v4 §2.2.g 的旁路采样验证 + 自适应 TTL：

```
5% kw_cache 命中 → 异步跑一遍完整归一化 → 比对结果 → 统计漂移率
漂移率 < 5%  → TTL 维持 3600s
漂移率 > 15% → TTL 降到 300s
```

这比 v3 的"固定 TTL 600s"好，因为它基于数据而非猜测做决策。

### 1.5 Embedding 模型版本管理

v4 §2.1.c 的 `model_ver` 入库 + 启动校验 + `reindex_keywords` 工具，是生产环境中 embedding 升级的必备基础设施。

## 2. v4 的缺陷

### 2.1 P0 级缺陷（设计层面，会导致实现者困惑或误用）

#### a) L1 / hot 双键拆分是过度设计

v4 §2.1.a 发现了一个真问题：L1 的 `task_sig` 不含 ctx，导致不同 ctx 的同 query 串数据。

但 v4 的修复方案引入了两套键：

```
apc:exact:<agent_id>:<task_sig>:<ctx_fp_hash>  # L1（"真正重复请求"）
apc:hot:<agent_id>:<task_sig>:<ctx_fp_hash>    # hot 层（"L2/L3 提升"）
```

**问题 1**：语义区分没有实际意义。一个请求是"真正重复"还是"语义等价后被提升"，对缓存使用者来说是透明且无关的。两者都需要同样的校验（版本、schema、TTL），都命中同一个模板。

**问题 2**：写路径缺失。v4 代码示例只展示了 `apc:exact` 的读取，没有展示写入。什么时候写入 `apc:exact`？第二次完全相同的 (query, ctx) 请求到来时？那第一次是怎么进入 `apc:hot` 的？

**问题 3**：增加了一次 Redis round-trip（查完 exact 查 hot）。实际收益为零——因为两者的 key 结构相同，只是前缀不同。

**v5 修正**：合二为一。只用一套键，ctx_fp 进 key：

```python
# L1: 精确匹配（query + ctx 都匹配）
l1_key = f"apc:tpl:{agent_id}:{task_sig}:{ctx_fp_hash}"
# 写入时机：L2/L3 命中后直接写 L1（就是 v3 的"提升"，不需要分两层）
# 读取时机：cache_lookup 的第一步
```

语义澄清：L1 不是"完全重复请求"，而是"相同 query + 相同 ctx 结构的请求"。这覆盖了"真正重复"（两次发送同一请求）和"语义等价被提升后第二次遇到类似请求"两个场景。不需要分开。

#### b) `build_candidates` 返回值设计有 API 异味

```python
# v4 代码片段
candidates = await build_candidates(query_vec, top_k=5)

if candidates is None:        # None = 强复用
    kw = candidates_top.keyword   # ← Bug！candidates 是 None，没有 .keyword 属性
elif candidates == []:        # []  = 强新建
    kw = await fallback_extract_keyword(query)
else:                         # list = 正常路径
    kw = await normalize_keyword(query, candidates)
```

**问题**：

- `None` 表示"强复用"但 `candidates_top` 本身不可用，需要额外的变量传递
- `None` vs `[]` vs `list` 三种返回类型，调用方必须记住各自的语义
- `candidates is None` 路径对 `candidates_top.keyword` 的引用直接 NameError

**v5 修正**：用 dataclass 或 NamedTuple 替代三态返回值：

```python
@dataclass
class CandidateResult:
    items: list[tuple[str, float]]         # 候选列表（可为空）
    action: Literal["shortcut_reuse",      # 强复用：items[0] 直接用作 keyword
                    "shortcut_new",         # 强新建：不调 LLM，走 fallback_extract
                    "ask_llm"]              # 让 LLM 决定

    @property
    def top_keyword(self) -> str | None:
        return self.items[0][0] if self.items else None

    @property
    def top_score(self) -> float | None:
        return self.items[0][1] if self.items else None
```

#### c) L1 写入时缺少 ctx_fp 的持久化

v4 §2.1.a 提出 L1 键加 ctx_fp_hash，但模板存入 Redis 时（template_gen），v4 没有说明模板元数据中是否也存了一份 ctx_fingerprint。这导致一个不对称：

- 查找时：计算当前 query 的 ctx_fp，用它做 L3 筛选和 L1 键
- 写入时：模板带的是当时 cache miss 请求的 ctx_fingerprint
- 问题：模板的 ctx_fingerprint 描述的是"生成这个模板的请求的上下文结构"，但之后命中这个模板的请求可能有不同的 ctx_structure 但被 L3 判定为兼容（因为 `ctx_compatible` 用的是区间/包含关系）

这意味着同一个模板可能被多个不同 ctx_fp 的请求命中，从而产生多个 L1 键指向同一个 template_id。当模板被驱逐时，需要清理所有这些 L1 键。

**v5 修正**：模板写入时维护反向索引：

```python
# template_gen 写入时：
# 维护 template_id → set of L1 keys 的反向映射
await redis.sadd(f"apc:tpl_refs:{tpl_id}", l1_key)
await redis.expire(f"apc:tpl_refs:{tpl_id}", ttl + 86400)  # 比模板多活一天

# 模板驱逐时：
refs = await redis.smembers(f"apc:tpl_refs:{tpl_id}")
if refs:
    await redis.delete(*refs)  # 级联清理所有 L1 键
await redis.delete(f"apc:tpl_refs:{tpl_id}")
```

### 2.2 P1 级缺陷（工程健壮性问题）

#### d) 多副本一致性方案过于复杂

v4 §2.2.d 推荐 pub/sub 失效推送方案。这个方案需要的代码：

- pub/sub 连接管理（断线重连、心跳）
- 订阅启动时的全量加载（先加载全量，再开始订阅，防止丢失中间消息）
- 消息序列化/反序列化
- 各副本独立维护本地 dict

而替代方案"每次从 Redis 拉全量"在 keyword < 5000 时真的有那么差吗？

**实际数字**：
- 5000 个 keyword，每个 384 维 float32 = 7.7MB
- Redis GET 一个 7.7MB 的 key，同机房延迟约 5-10ms
- 每个 cache_lookup 只多 5-10ms（且只在 kw_cache miss 时才需要向量搜索）

对比 pub/sub 方案：
- pub/sub 方案也需要定期从 Redis 拉全量（启动时 + 断线重连后）
- 正常运行时不拉全量，但需要处理消息积压、乱序、重复
- 代码量约 150-200 行（连接管理 + 消息处理 + 本地 index 维护）

**v5 修正**：简化方案分两阶段：

```
阶段 A（keyword < 2000）：每次从 Redis 拉全量（一个 GET，5-10ms）
阶段 B（keyword ≥ 2000）：Redis Sorted Set 按时间戳增量拉取
```

阶段 B 的实现：

```python
# 写入 keyword_index 时记录时间戳
ZADD apc:kw_timeline <timestamp> <keyword>

# 各副本维护上次同步时间戳，定期拉增量
last_sync_ts = local_last_sync_ts
new_kws = await redis.zrangebyscore("apc:kw_timeline", last_sync_ts, "+inf")
for kw in new_kws:
    local_index[kw] = await fetch_kw_embedding(kw)
local_last_sync_ts = time.time()
```

这比 pub/sub 简单得多：不需要连接管理、不需要处理消息语义、不需要启动时的全量同步逻辑。唯一的代价是最多一个同步间隔（比如 5 秒）的最终一致性窗口。对于 keyword 索引来说，5 秒的窗口完全可接受——远小于 kw_cache 的 TTL（600s）。

#### e) 微批处理 EmbedBatcher 可能适得其反

v4 §2.2.e 设计了 `EmbedBatcher`，用 5ms 窗口累积 encode 请求做批处理。但：

- 批处理只在并发 encode 请求时才有效。APC 的 keyword_node 是在 cache_lookup 中调用的，且只在 kw_cache miss 时。在稳定状态下（命中率 > 40%），大部分请求根本不会走到 encode。
- 批处理引入了 5ms 的人为延迟（等待窗口填满）。
- 当只有一个请求在等待时，5ms 后批大小 = 1，批处理没有任何收益，纯浪费 5ms。

**v5 修正**：不用批处理，只用信号量 + 专用 executor：

```python
EMBED_EXECUTOR = ThreadPoolExecutor(max_workers=2)
EMBED_SEMAPHORE = asyncio.Semaphore(4)

async def embed(text: str) -> np.ndarray:
    async with EMBED_SEMAPHORE:
        loop = asyncio.get_event_loop()
        return await loop.run_in_executor(EMBED_EXECUTOR, model.encode, text)
```

信号量 4 意味着最多 4 个并发 encode。2 个 worker 线程足够——PyTorch 内部已经有并行，多 worker 争 GIL 反而降低吞吐。这个方案比批处理简单，且不会引入人为延迟。

如果未来确实需要批处理（encode 成为瓶颈且并发量大），可以在 `embed()` 的调用方层面做（而不是在 encode 方法内部），比如 keyword_node 一次性 encode 多个候选 keyword 的 embedding。但目前不需要。

#### f) 归一化回退链中 `retry_with_stricter_prompt` 可能导致请求超时

v4 §2.1.b 的反实体注入逻辑：

```python
if contains_query_entities(keyword, query):
    keyword = await retry_with_stricter_prompt(query, candidates)
```

如果第二次调用也失败（LLM 再次返回实体），没有第三次回退。应该给重试一次上限，然后 fallback：

```python
if contains_query_entities(keyword, query):
    keyword = await retry_with_stricter_prompt(query, candidates)
    if contains_query_entities(keyword, query):
        # 不无限重试，取 sanitize 后的结果
        keyword = sanitize_keyword(keyword)
```

### 2.3 P2 级缺陷（设计完整性问题）

#### g) 冷启动策略的完全缺失

v1 有 `/cache/warmup` 端点。v2/v3/v4 都没提。冷启动对 keyword 向量索引的影响：

- 服务首次启动，keyword_index 为空
- 所有请求都走"强新建短路"（候选为空）→ `fallback_extract_keyword`
- 每个 cache miss 写入一个新 keyword
- keyword_index 从 0 开始积累，归一化效果从零逐步增强

这不是 bug，但会导致上线初期的命中率显著低于稳态。v4 应该给出冷启动缓解策略：

```python
# 方案 A：从 PG / S3 恢复 keyword_index（如果有持久化快照）
async def restore_keyword_index():
    if await redis.exists("apc:kw_index_state"):
        # Redis 中有数据（可能从 AOF 恢复），不需要重建
        return
    # 从对象存储恢复
    snapshot = await s3.get("apc/kw_index_snapshot.json")
    if snapshot:
        for kw_entry in snapshot["keywords"]:
            await redis.hset(f"apc:kw_meta:{kw_entry['keyword']}",
                             mapping=kw_entry)
    # 如果没有快照，keyword_index 从空开始（接受冷启动）

# 方案 B：定期快照 keyword_index 到对象存储
async def snapshot_keyword_index():
    keywords = []
    for key in await redis.keys("apc:kw_meta:*"):
        kw = key.decode().split(":", 2)[-1]
        meta = await redis.hgetall(key)
        keywords.append({"keyword": kw, **meta})
    await s3.put("apc/kw_index_snapshot.json", json.dumps({"keywords": keywords}))
```

#### h) 两个并发 cache miss 可能创建语义重复的 keyword

场景：

```
t=0: 两个并发请求同时到达不同副本
  Req A: "Calculate working capital ratio"  → kw_index 中有 {"working_capital"}
  Req B: "Compute WC ratio for company"     → kw_index 中有 {"working_capital"}

t=1: 两个请求各自找到候选 {"working_capital"}，distance 都在 0.7 左右
  Req A: LLM 判断"语义相似但需要更具体的关键词" → 新建 "working_capital_ratio"
  Req B: LLM 判断"语义相似但措辞不同"         → 新建 "wc_ratio_calculation"

t=2: 两个请求都写入 keyword_index
  kw_index 现在有 {"working_capital", "working_capital_ratio", "wc_ratio_calculation"}
  但实际上后两个应该是同一个
```

**v5 修正**：写入 keyword 前加分布式锁 + 二次检查：

```python
async def upsert_keyword(keyword: str, embedding: np.ndarray):
    # 先检查是否已存在（另一个并发请求可能刚好写入）
    if await redis.hexists("apc:kw_meta", keyword):
        return  # 已存在，不需要重复写入

    # 用分布式锁保护"检查 → 写入"窗口
    lock_key = f"apc:kw_lock:{hash(keyword)}"
    acquired = await redis.set(lock_key, "1", nx=True, ex=10)
    if not acquired:
        # 另一个请求正在写入同一个 keyword，等待
        await asyncio.sleep(0.5)
        return

    try:
        # 二次检查
        if await redis.hexists("apc:kw_meta", keyword):
            return

        # 额外检查：是否有非常相近的 keyword 同时被写入
        #（简单的向量搜索，看是否有人在短时间内写入相似 keyword）
        recent = await search_keyword_index(embedding, top_k=2)
        if recent and recent[0][1] > 0.95:
            # 已有高度相似的 keyword，日志告警但不阻塞
            logger.warning("near_duplicate_keyword",
                           new=keyword, existing=recent[0][0])
        await write_kw_meta(keyword, embedding)
    finally:
        await redis.delete(lock_key)
```

这不能完全阻止（LLM 的"判断"本质上是非确定性的），但至少能检测并记录，供后续别名合并。

#### i) 归一化决策日志的 PG Schema 缺失

v4 §2.3.i 说收集 `(query_embed, candidates_embed, decision)`，但没给 schema。没有 schema 的数据收集等于没收集——因为训练时需要重新清洗。

```sql
CREATE TABLE kw_normalization_log (
    id          BIGSERIAL PRIMARY KEY,
    query_hash  TEXT NOT NULL,          -- 原始 query 的 hash
    query_vec   VECTOR(384) NOT NULL,   -- pgvector 类型
    candidates  JSONB NOT NULL,         -- [{keyword, distance}, ...]
    action      TEXT NOT NULL,          -- 'reuse_candidate' | 'new_keyword'
    chosen_kw   TEXT NOT NULL,
    chosen_rank INT,                    -- 如果复用，排名第几（NULL=新建）
    model_ver   TEXT NOT NULL,          -- EMBED_MODEL_VER
    latency_ms  INT NOT NULL,           -- 归一化耗时
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_kw_log_query_hash ON kw_normalization_log(query_hash);
CREATE INDEX idx_kw_log_action ON kw_normalization_log(action);
```

### 2.4 P3 级缺陷（小问题）

#### j) `retry_with_stricter_prompt` 始终未给出实现

v4 §2.1.b 提到但未定义。给出参考：

async def retry_with_stricter_prompt(query, candidates):

## 3. v5 优化汇总

### 3.1 核心修正（P0）

| # | 问题 | v4 做法 | v5 修正 |
|---|------|---------|---------|
| 1 | L1/hot 双键过度设计 | 两套键 + 两套语义 | 合并为单层 L1，key 含 ctx_fp_hash |
| 2 | build_candidates 三态返回值 | None / [] / list | `CandidateResult` dataclass |
| 3 | template 驱逐时 L1 键残留 | 未处理 | 反向索引 `apc:tpl_refs` 级联清理 |

### 3.2 工程简化（P1）

| # | 问题 | v4 做法 | v5 修正 |
|---|------|---------|---------|
| 4 | 多副本一致性 | pub/sub 失效推送 | ZSET 时间戳增量同步（更简单） |
| 5 | encode 批处理 | EmbedBatcher + 5ms 窗口 | 信号量 + 专用 executor（无人工延迟） |
| 6 | 反实体注入二重试 | 无限制 | 最多 2 次，失败后 sanitize 兜底 |

### 3.3 补全（P2）

| # | 问题 | v5 修正 |
|---|------|---------|
| 7 | 冷启动无策略 | keyword_index 定期快照到对象存储 + 启动恢复 |
| 8 | 并发写入语义重复 keyword | 分布式锁 + 二次检查 + 近重复检测 |
| 9 | 决策日志无 schema | pgvector schema + 索引 |

### 3.4 继承 v4 不做修改的部分

v4 §6 的 5 条范围控制全部继承。外加：

- **不对 keyword 做自动语义聚类/合并**（LLM 驱动的合并会引入不可控的假阳性，走人工别名 + 管理工具）
- **不引入分布式事务**（keyword → template 的一致性用最终一致性 + 旁路修复脚本处理）
- **不引入 pgvector 做向量搜索**（pgvector 只在决策日志存储时用 VECTOR 类型方便后续分析，不做在线查询）

### 4.1 简化后的 cache_lookup（合并 exact/hot）

```python
async def cache_lookup_v5(state: AgentState) -> AgentState:
    agent_id   = state.get("agent_id", "default")
    tools_hash = state.get("tools_hash", "default")
    query      = state["query"]
    ctx        = state["context"]

    # L1: 精确匹配（query + ctx_fp，合并了 v4 的 exact + hot）
    task_sig = sha256(
        f"{normalize(query)}|{agent_id}|{tools_hash}".encode()
    ).hexdigest()[:16]
    ctx_fp = compute_fingerprint(ctx, state.get("tools", []), agent_id)
    ctx_fp_hash = sha256(
        json.dumps(ctx_fp, sort_keys=True, default=str).encode()
    ).hexdigest()[:12]

    l1_key = f"apc:l1:{agent_id}:{task_sig}:{ctx_fp_hash}"
    if tpl_id := await redis.get(l1_key):
        if tpl := await load_and_validate_template(tpl_id, tools_hash):
            cache_hits_total.labels(layer="L1").inc()
            return _hit(state, tpl, layer="L1")

    # L2/L3: keyword + ctx 筛选
    state = await keyword_node_v5(state)
    keyword = state["keyword"]
    if not keyword:
        return {**state, "cache_hit": False}

    candidates = await redis.smembers(
        f"apc:tpl_idx:{agent_id}:{keyword}"
    )
    if not candidates:
        return {**state, "cache_hit": False}

    for tpl_id in list(candidates)[:MAX_CANDIDATES_TO_CHECK]:
        tpl = await load_and_validate_template(tpl_id, tools_hash)
        if tpl and ctx_compatible(
            tpl.get("ctx_fingerprint", {}), ctx_fp
        ):
            # 提升到 L1（单一键，不区分 exact/hot）
            await redis.set(l1_key, tpl_id, ex=86400)
            # 维护反向索引（驱逐时级联清理）
            await redis.sadd(
                f"apc:tpl_refs:{tpl_id}", l1_key
            )
            cache_hits_total.labels(layer="L2_L3").inc()
            return _hit(state, tpl, layer="L2_L3")

    return {**state, "cache_hit": False}
```

### 4.2 简化后的 keyword_node（自适应候选 + 无复杂批处理）

```python
async def keyword_node_v5(state: AgentState) -> AgentState:
    query = state["query"]
    qhash = sha256(normalize(query).encode()).hexdigest()[:16]

    # kw_cache
    cached = await redis.get(f"apc:kw_cache:{qhash}")
    if cached:
        kw = cached.decode()
        # 旁路漂移采样（少量采样，异步不阻塞）
        if random.random() < DRIFT_SAMPLE_RATE:
            asyncio.create_task(_check_drift(query, kw))
        return {**state, "keyword": kw}

    # 向量召回 + 自适应短路
    query_vec = await embed(query)
    result = await build_candidates_v5(query_vec, top_k=5)

    if result.action == "shortcut_reuse":
        kw = result.top_keyword
    elif result.action == "shortcut_new":
        kw = await fallback_extract_keyword(query)
    else:  # "ask_llm"
        kw = await normalize_keyword(query, result.items)

    kw = sanitize_keyword(kw)
    ttl = current_kw_cache_ttl()
    await redis.setex(f"apc:kw_cache:{qhash}", ttl, kw)

    # 异步记录决策日志
    asyncio.create_task(_log_decision(query, query_vec, result, kw))

    return {**state, "keyword": kw}
```

### 4.3 CandidateResult + 自适应候选构造

```python
@dataclass
class CandidateResult:
    items: list[tuple[str, float]]
    action: str  # "shortcut_reuse" | "shortcut_new" | "ask_llm"

    @property
    def top_keyword(self) -> str | None:
        return self.items[0][0] if self.items else None


DIST_HIGH = 0.85
DIST_LOW  = 0.50


async def build_candidates_v5(
    query_vec: np.ndarray, top_k: int = 5
) -> CandidateResult:
    raw = await knn_search(query_vec, top_k=top_k)
    filtered = [(kw, score) for kw, score in raw if score >= DIST_LOW]

    if not filtered:
        return CandidateResult(items=[], action="shortcut_new")

    if len(filtered) == 1 and filtered[0][1] >= DIST_HIGH:
        return CandidateResult(items=filtered, action="shortcut_reuse")

    if (len(filtered) >= 2
        and filtered[0][1] >= DIST_HIGH
        and filtered[0][1] - filtered[1][1] > 0.15):
        return CandidateResult(items=[filtered[0]], action="shortcut_reuse")

    return CandidateResult(items=filtered[:3], action="ask_llm")
```

### 4.4 多副本一致性（ZSET 增量同步替代 pub/sub）

```python
# 全局配置
KW_INDEX_SYNC_INTERVAL = 5  # 秒


class KeywordIndexManager:
    """各 API 副本本地维护的 keyword embedding 缓存。"""

    def __init__(self, redis_client):
        self._redis = redis_client
        self._index: dict[str, np.ndarray] = {}  # keyword → embedding
        self._last_sync_ts = 0.0
        self._sync_task: asyncio.Task | None = None

    async def start(self):
        # 启动时全量加载
        await self._full_reload()
        # 定时增量同步
        self._sync_task = asyncio.create_task(self._periodic_sync())

    async def _full_reload(self):
        keys = await self._redis.keys("apc:kw_meta:*")
        new_index = {}
        for key in keys:
            kw = key.decode().split(":", 2)[-1]
            meta = await self._redis.hgetall(key)
            if meta[b"model_ver"].decode() == EMBED_MODEL_VER:
                new_index[kw] = np.frombuffer(
                    meta[b"embedding"], dtype=np.float32
                )
        self._index = new_index
        self._last_sync_ts = time.time()

    async def _periodic_sync(self):
        while True:
            await asyncio.sleep(KW_INDEX_SYNC_INTERVAL)
            try:
                new_kws = await self._redis.zrangebyscore(
                    "apc:kw_timeline", self._last_sync_ts, "+inf"
                )
                for kw in new_kws:
                    kw = kw.decode() if isinstance(kw, bytes) else kw
                    # 只拉取版本匹配且本地没有的
                    if kw not in self._index:
                        meta = await self._redis.hgetall(f"apc:kw_meta:{kw}")
                        if meta and meta[b"model_ver"].decode() == EMBED_MODEL_VER:
                            self._index[kw] = np.frombuffer(
                                meta[b"embedding"], dtype=np.float32
                            )
                self._last_sync_ts = time.time()
                # 清理已删除的 keyword（通过对比 Redis keyset，低频做）
            except Exception:
                logger.warning("kw_index_sync_failed", exc_info=True)

    async def search(self, query_vec, top_k=5):
        # 在本地 index 做 KNN
        if not self._index:
            return []
        items = list(self._index.items())
        scores = [
            (kw, float(cosine_similarity(query_vec, vec)))
            for kw, vec in items
        ]
        scores.sort(key=lambda x: -x[1])
        return scores[:top_k]

    async def on_keyword_written(self, keyword: str):
        """由 template_gen 调用的钩子：新 keyword 写入 Redis 后更新本地。"""
        # 直接更新本地（不等下一个 sync 周期），保证写入副本立即可见
        meta = await self._redis.hgetall(f"apc:kw_meta:{keyword}")
        if meta and meta[b"model_ver"].decode() == EMBED_MODEL_VER:
            self._index[keyword] = np.frombuffer(
                meta[b"embedding"], dtype=np.float32
            )
```

**对比 pub/sub 方案**：
- 不需要管理 pub/sub 连接生命周期
- 不需要处理消息丢失/重复/乱序
- 代码量减少约 40%
- 代价：最多 5 秒的最终一致性窗口（可接受）

### 4.5 template 驱逐的级联清理

```python
async def evict_template(tpl_id: str):
    """驱逐模板时级联清理所有关联的 L1 键和索引。"""
    # 1. 清理反向索引指向的所有 L1 键
    refs = await redis.smembers(f"apc:tpl_refs:{tpl_id}")
    if refs:
        await redis.delete(*refs)

    # 2. 从 tpl_idx 移除
    keywords = await redis.keys(f"apc:tpl_idx:*")
    for kw_key in keywords:
        await redis.srem(kw_key, tpl_id)
        # 清理空 keyword 索引
        if await redis.scard(kw_key) == 0:
            keyword = kw_key.decode().split(":", 2)[-1]
            await redis.delete(kw_key)
            await redis.hdel("apc:kw_meta", keyword)
            # 更新 timeline（注意这里不是删除，而是标记已删除，让各副本同步感知）
            await redis.zrem("apc:kw_timeline", keyword)
            await redis.zadd("apc:kw_tombstone", {keyword: time.time()})

    # 3. 删除模板本体
    await redis.hdel("apc:tpl", tpl_id)
    await redis.delete(f"apc:tpl_refs:{tpl_id}")

    # 4. 清理 hot 层（如果有残留）
    hot_keys = await redis.keys(f"apc:hot:*:{tpl_id}")
    if hot_keys:
        await redis.delete(*hot_keys)
```

## 5. v4 → v5 差异对照

| 维度 | v4 | v5 |
|------|----|----|
| L1 缓存结构 | exact + hot 双键 | 单层 L1（合并，key 含 ctx_fp） |
| 候选返回值 | `None` / `[]` / `list` 三态 | `CandidateResult` dataclass |
| 模板驱逐 | 未处理 L1 残留 | 反向索引 `apc:tpl_refs` 级联清理 |
| 多副本一致性 | pub/sub 失效推送 | ZSET 增量同步（更简单） |
| encode 治理 | 微批处理 + 5ms 窗口 | 信号量 + 专用 executor |
| 冷启动 | 无策略 | keyword_index S3 快照 + 启动恢复 |
| 并发 keyword 写入 | 无保护 | 分布式锁 + 近重复检测 |
| 决策日志 | 未定义 schema | pgvector schema + 索引 |
| 反实体注入重试 | 无限/未定义 | 最多 2 次，失败后 sanitize 兜底 |
